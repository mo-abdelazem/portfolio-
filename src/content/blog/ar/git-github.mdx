Git هو الأداة التي يستخدمها كل مطوّر كل يوم، والتي يتعلّم معظم الناس ما يكفي منها للتدبّر — حتى يأتي يوم سيّئ بتغيير ضائع أو فرع متشابك. فهم *النموذج* تحت الأوامر يحوّل Git من مجموعة تعويذات مخيفة إلى شيء تستطيع التفكير فيه. هذا المقال يبني ذلك النموذج — المناطق الثلاث، والفروع، والريموت — وسير عمل GitHub المبنيّ فوقه. (يقترن طبيعيًّا بـ [وحدات ES والأدوات](/blog/es-modules)، لأن `package.json` ومستودعك يعيشان معًا.)

> نموذج Git كلّه **لقطات عبر الزمن**. الالتزام (commit) لقطة لمشروعك تستطيع العودة إليها دائمًا؛ والفرع مؤشّر متحرّك إلى خطّ من اللقطات. وحالما ترى الالتزامات كنقاط حفظ والفروع كخطوط زمنية متوازية، تتوقّف الأوامر عن كونها سحرًا.

## ما هو التحكّم في الإصدارات

التحكّم في الإصدارات يسجّل تاريخ مشروعك — كل تغيير، ومن أجراه، ولماذا — كي تراجعه، وتتراجع عن الأخطاء، وتتعاون دون أن يطمس أحدكم عمل الآخر. **Git** هو نظام التحكّم في الإصدارات (يعمل محليًّا)؛ و**GitHub** خدمة تستضيف مستودعات Git على الإنترنت للمشاركة والتعاون. مترابطان لكنهما متمايزان: تستطيع استخدام Git بلا GitHub إطلاقًا.

## المناطق الثلاث

هذا النموذج الذهني الذي يجعل Git مفهومًا. يتحرّك الملفّ عبر ثلاثة أماكن:

- **دليل العمل (working directory)** — ملفّاتك الفعلية، حيث تحرّر.
- **منطقة التهيئة (staging/index)** — منطقة احتجاز للتغييرات التي تريدها في التزامك *التالي*.
- **المستودع (repository)** — التاريخ الملتزَم، اللقطات الدائمة.

```bash
git init              # ابدأ تتبّع مشروع (ينشئ المستودع)
# ...حرّر الملفّات في دليل العمل...
git add file.js       # هيّئ تغييرًا (دليل العمل → التهيئة)
git commit -m "msg"   # سجّل التغييرات المهيّأة (التهيئة → المستودع)
```

خطوتا `add` ثم `commit` مقصودتان: التهيئة تتيح صياغة التزام من التغييرات التي تختارها بالضبط، بدل كل شيء دفعة.

## التهيئة والالتزام

الحلقة اليومية. افحص ما تغيّر، هيّئه، التزمه برسالة واضحة:

```bash
git status            # ما تغيّر، ما هُيّئ
git diff              # تغييرات الأسطر بالضبط (غير المهيّأة)
git add .             # هيّئ كل ما تغيّر
git add file.js       # أو هيّئ ملفًّا واحدًا
git commit -m "Add login validation"
git log --oneline     # تصفّح التاريخ
```

الالتزام الجيّد **صغير ومركّز**، برسالة تقول *لماذا*، بصيغة الأمر ("Fix overflow bug"، لا "fixed stuff"). التزِم كثيرًا — كل التزام نقطة حفظ تستطيع العودة إليها.

## الفروع

**الفرع (branch)** مؤشّر متحرّك إلى خطّ من الالتزامات — يتيح العمل على ميزة دون إزعاج الشيفرة الرئيسية، ثم دمجها عائدةً عند الجهوزية:

```bash
git branch                    # اسرد الفروع
git switch -c feature/login   # أنشئ وانتقل إلى فرع جديد
# ...التزِم العمل على الفرع...
git switch main               # عُد إلى main
git merge feature/login       # أحضِر التزامات الميزة إلى main
git branch -d feature/login   # احذف الفرع المدموج
```

التفرّع رخيص وسريع في Git، ولهذا سير العمل المعتاد **فرع لكل ميزة أو إصلاح** — يبقى main مستقرًّا، ويحدث العمل على الفروع، ويُدمَج العمل المنتهي.

## تعارضات الدمج

حين يغيّر فرعان *الأسطر نفسها*، لا يستطيع Git القرار تلقائيًّا ويبلّغ عن **تعارض (conflict)**. يعلّم الموضع في الملفّ:

```text
<<<<<<< HEAD
const timeout = 3000;
=======
const timeout = 5000;
>>>>>>> feature/login
```

تحلّه بتحرير الملفّ إلى النسخة التي تريدها (مزيلًا علامات `<<<`/`===`/`>>>`)، ثم التهيئة والالتزام:

```bash
# حرّر الملفّ لإبقاء الشيفرة الصحيحة
git add file.js
git commit            # يُكمِل الدمج
```

التعارضات تبدو مخيفة لكنها روتينية — تعني فقط أن شخصين لمسا الأسطر نفسها، وGit يطلب *منك* الاختيار.

## `.gitignore`

بعض الملفّات لا ينبغي تتبّعها — التبعيات، ومخرجات البناء، والأسرار، ونفايات نظام التشغيل. ملفّ **`.gitignore`** يسرد الأنماط التي يتخطّاها Git:

```text
node_modules/
dist/
.env
.DS_Store
*.log
```

لا تلتزم أبدًا `node_modules/` (قابل لإعادة التثبيت من `package.json`) أو ملفّات `.env` (تحمل أسرارًا). أضف قواعد التجاهل *قبل* أول التزام حيثما أمكن — حالما يُتتبَّع ملفّ، يأخذ تجاهله لاحقًا خطوة إضافية.

## الريموت: Push وPull

**الريموت (remote)** نسخة من مستودعك مستضافة في مكان آخر (عادةً على GitHub). تزامن التزاماتك المحلّية معه:

```bash
git clone https://github.com/user/repo.git   # انسخ مستودعًا بعيدًا محليًّا
git remote add origin <url>                    # اربط مستودعًا محليًّا موجودًا بريموت
git push origin main                           # أرسِل التزاماتك للأعلى
git pull origin main                           # اجلب التزامات الآخرين وادمجها
git fetch                                       # نزّل التغييرات البعيدة دون دمج
```

`origin` هو الاسم الاصطلاحي لريموتك الرئيسي. **Push** يرفع التزاماتك؛ و**pull** ينزّل ويدمج التزامات الآخرين. اسحب (pull) قبل أن تبدأ العمل كي تبني على الأحدث، وادفع (push) حين تملك شيئًا يستحقّ المشاركة.

## سير عمل طلبات السحب في GitHub

هكذا تتعاون الفرق (والمصادر المفتوحة). الدورة:

1. **تفرّع** عن main لتغييرك (`git switch -c fix/typo`).
2. **التزِم** عملك و**ادفع** الفرع إلى GitHub.
3. افتح **طلب سحب (pull request / PR)** — اقتراح لدمج فرعك في main، حيث يراجع الآخرون الفرق، ويعلّقون، ويوافقون.
4. حالما يُوافَق (وتجتاز الفحوص)، **ادمج** الـ PR؛ ويُحذَف الفرع.

للمستودعات التي لا تملك صلاحية الكتابة فيها، **تشعّبها (fork)** (نسختك على GitHub)، تدفع إلى تشعّبك، وتفتح PR منه. الـ PR هو حيث تحدث مراجعة الشيفرة والنقاش والاختبارات الآلية قبل وصول الشيفرة إلى main — قلب التطوير التعاوني.

## الأخطاء الشائعة

- التزام `node_modules/` أو مخرجات البناء أو أسرار `.env` — أضف `.gitignore` أولًا.
- رسائل التزام غامضة ("update"، "fix") لا تقول شيئًا عن *لماذا* — اكتب النيّة.
- التزامات ضخمة تخلط تغييرات كثيرة غير مترابطة — أبقِها صغيرة ومركّزة.
- العمل مباشرةً على `main` بدل فرع، فيصعب مراجعة التغييرات والتراجع عنها.
- نسيان `pull` قبل البدء، ثم الاصطدام بتعارضات يمكن تجنّبها.
- الهلع عند تعارض دمج — إنه روتيني؛ فقط حرّر وهيّئ والتزِم.
- الدفع القسري (`git push --force`) إلى فرع مشترك، فطمس عمل الآخرين.
- الخلط بين Git (الأداة المحلّية) وGitHub (المضيف) — تستطيع استخدام Git كليًّا دون اتصال.

## تمارين

جرّب كلًّا منها قبل فتح الحل.

### تمرين 1 — ابدأ واحفظ

لديك مجلّد مشروع جديد. هيّئ Git واصنع أول التزام.

<details>

<summary>إظهار الحل</summary>

```bash
git init
git add .
git commit -m "Initial commit"
```

`init` ينشئ المستودع، و`add .` يهيّئ كل الملفّات الحالية إلى منطقة التهيئة، و`commit` يسجّلها كأول لقطة.

</details>

### تمرين 2 — فرع ميزة

أنشئ فرعًا لميزة جديدة، عالمًا أنك ستدمجه لاحقًا.

<details>

<summary>إظهار الحل</summary>

```bash
git switch -c feature/search
# ...حرّر، git add، git commit...
git switch main
git merge feature/search
```

`switch -c` ينشئ الفرع وينتقل إليه؛ تلتزم العمل هناك، ثم تدمجه عائدًا إلى `main` — يبقى main نظيفًا أثناء بنائك.

</details>

### تمرين 3 — تجاهل الأشياء الصحيحة

ما الذي ينتمي إلى `.gitignore` لمشروع Node/ويب نموذجيّ، ولماذا؟

<details>

<summary>إظهار الحل</summary>

```text
node_modules/   # قابل لإعادة التثبيت من package.json — ضخم وبلا جدوى لتتبّعه
dist/           # مخرجات البناء — مولَّدة، لا مصدر
.env            # أسرار — لا تلتزمها أبدًا
.DS_Store       # نفايات نظام التشغيل
```

القاعدة: تجاهل أي شيء مولَّد أو قابل لإعادة التثبيت أو سرّيّ. ملفّات المصدر و`package.json` هي ما تتتبّعه؛ وكل ما يُشتقّ منها لا.

</details>

### تمرين 4 — شارِك عملك

التزمت محليًّا وتريده على GitHub ليراجعه زميل. ما السير؟

<details>

<summary>إظهار الحل</summary>

```bash
git switch -c fix/header        # فرع
# ...التزِم عملك...
git push origin fix/header      # ادفع الفرع إلى GitHub
# ثم افتح طلب سحب على GitHub للمراجعة
```

ادفع الفرع (لا مباشرةً إلى main)، ثم افتح PR كي يُراجَع التغيير ويُناقَش قبل الدمج — سير التعاون المعياريّ.

</details>

## النموذج الذهني الذي تحتفظ به

Git **لقطات عبر الزمن**. تتدفّق الملفّات عبر ثلاث مناطق — **دليل العمل → التهيئة (`add`) → المستودع (`commit`)** — وكل التزام نقطة حفظ تستطيع العودة إليها. و**الفرع** مؤشّر متحرّك إلى خطّ من الالتزامات، فسير العمل اليوميّ *فرع لكل ميزة، التزِم صغيرًا وكثيرًا، ادمج عائدًا* — و**التعارض** مجرّد Git يطلب منك الاختيار حين لمس فرعان الأسطر نفسها. و**الريموت** (GitHub) نسخة مستضافة تـ**دفع** إليها وتـ**سحب** منها، و**طلب السحب** حيث يحدث التعاون: تفرّع، ادفع، افتح PR، راجع، ادمج. أبقِ الأسرار والملفّات المولَّدة خارجًا بـ **`.gitignore`**، واكتب رسائل التزام تشرح *لماذا*، فيتوقّف Git عن كونه مصدر رهبة ويصير شبكة الأمان التي تتيح لك العمل بلا خوف.
